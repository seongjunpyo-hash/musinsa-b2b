# 비즈니스 규칙

## 1. 가격 / 환율

### 1-1. 홀세일가 계산
파트너가 보는 USD 가격 = 한국 소매가 × (1 - 파트너 할인율) ÷ 환율

```javascript
function calculateWholesalePrice(retailPriceKrw, discountRate, krwPerUsd) {
  const usdPrice = (retailPriceKrw / krwPerUsd) * (1 - discountRate);
  return Math.floor(usdPrice * 100) / 100; // 소수점 둘째 자리에서 버림
}
```

**예시**
- retail_price_krw = 89,000원
- discount_rate = 0.30 (30%)
- krw_per_usd = 1350
- → wholesale_usd = floor((89000 / 1350) × 0.70 × 100) / 100 = `46.14 USD`

### 1-2. 할인된 한국 소매가 (참고용 표시)
홀세일가 모달의 `홀세일가 (원)` 컬럼 = `retail_price_krw × (1 - discount_rate)`

### 1-3. 환율 적용 시점
- **주문 생성 시점의 환율로 unit_price 고정** → 이후 환율 변경되어도 기존 주문 금액은 변하지 않음
- 카탈로그/장바구니 표시는 **현재 환율** 적용

### 1-4. 환율 업데이트
- 어드민이 수동 업데이트
- 변경 이력 기록 권장 (감사/회계 목적)
- 변경 시점 기준으로 새로운 주문에 적용

## 2. 주문 (Order)

### 2-1. 주문 상태 전이
```
[주문접수]
   │
   ├─ 어드민 승인 ──► [주문완료]
   │
   ├─ 어드민 반려 ──► [반려]
   │                    │
   │                    └─ (Admin이 suggested_items 첨부 가능)
   │                    └─ 파트너가 제안안으로 신규 주문 가능
   │
   └─ 파트너 취소 ──► [주문취소]
                        (주문접수 상태에서만 가능)
```

### 2-2. 상태별 가능 액션

| 상태 | 어드민 가능 | 파트너 가능 |
|---|---|---|
| 주문접수 | 승인, 반려, 엑셀 다운로드 | 취소, 상세 보기 |
| 주문완료 | 엑셀 다운로드 | 상세 보기 (취소 불가) |
| 반려 | (없음) | 상세 보기, 제안안 재주문 |
| 주문취소 | (없음) | 상세 보기 |

### 2-3. 일괄 처리 가능 조건
- **일괄 승인**: 선택된 주문 중 `status: 주문접수`인 것만
- **일괄 반려**: 선택된 주문 중 `status: 주문접수`인 것만
- **일괄 엑셀 다운로드**: 선택된 주문 중 `status: 주문완료`인 것만
- 혼합 상태 선택 시 처리 가능한 것만 진행하며 모달로 안내 (예: "선택한 5건 중 3건이 주문접수 상태입니다. 해당 3건만 처리됩니다")

### 2-4. 주문 생성 검증
- items 배열 비어있을 수 없음
- 각 item의 productId가 실제 존재하는 visible 상품이어야 함
- 각 item의 color는 해당 상품의 colors_list에 있어야 함
- 각 item의 size는 해당 상품의 sizes에 있어야 함
- qty > 0 (양의 정수)
- 비활성 파트너는 주문 불가
- 수주회 미진행 기간엔 주문 불가 (`order_settings.enabled === false`)

### 2-5. 주문 ID 생성
- 형식: `ORD-{YYYY}-{NNNN}`
- YYYY: 주문 생성 연도
- NNNN: 해당 연도 내 순번 (4자리 0패딩)
- 예: `ORD-2026-0001`, `ORD-2026-0042`

### 2-6. 반려 시 제안안 (suggested_items)
- 어드민이 반려 사유 + 수정 제안안을 함께 보낼 수 있음
- 파트너 측 My Orders 상세에서 `Updated Order Suggested by Admin` 영역으로 노출
- 파트너가 "GO TO ORDER CONFIRMATION" 클릭 시 제안안 항목들이 장바구니에 자동 로드되고 Order Confirmation 페이지로 이동

## 3. 파트너 (Partner)

### 3-1. 등록 검증
- `company_name` 중복 불가 (대소문자 구분 없이)
- `email` 중복 불가 (대소문자 구분 없이)
- `discount_rate`: 1~100 사이 정수 (% 단위로 입력) → DB에는 0.01~1.00 소수로 저장
- 빈 값 / 공백만 입력 불가

### 3-2. 활성/비활성
- `is_active: false` → 로그인 차단
- 비활성 파트너의 기존 주문은 그대로 유지 (조회 가능)
- 비활성 → 활성으로 토글 가능 (어드민)

### 3-3. 삭제
- 어드민이 hard delete 가능
- 운영 권장: soft delete (`is_active=false` + `deleted_at` 기록) — 과거 주문 참조 무결성 유지
- 삭제된 파트너의 주문은 partner_name이 그대로 남아있음 (denormalized field)

### 3-4. 파트너 ID 생성
- 형식: `PTR-{NNN}` (3자리 0패딩)
- 자동 증가 (max + 1)
- 중복 발생 시 다음 사용 가능 번호 자동 할당

### 3-5. 파트너별 할인율
- 파트너마다 다른 할인율 가질 수 있음
- 동일 상품도 파트너마다 다른 홀세일가가 산출됨
- 할인율 변경 시 모든 신규 주문에 적용 (기존 주문은 영향 없음)

## 4. 상품 (Product)

### 4-1. 상품 ID 생성
- 형식: `P-{NNN}` (3자리 0패딩)
- 자동 증가 (max + 1)

### 4-2. 등록 / 수정 검증
- 필수: id, style_no, name, color, material, origin, category, retail_price_krw, ship_date, season, gender, sizes, images
- `style_no`: unique 권장 (현재는 중복 가능)
- `retail_price_krw`: 양의 정수
- `images`: 최소 1개 (최대 권장 5~10개)
- `sizes`: 최소 1개

### 4-3. 노출 제어 (`is_visible`)
- `true` 만 파트너 포털에서 보임
- 어드민 페이지에서는 모두 노출
- 일괄 노출/비노출 가능

### 4-4. HERO 상품 (`is_hero`)
- 파트너 메인 페이지의 HERO 섹션 자동 후보
- 전시 설정에서 명시적으로 선정한 상품이 우선

### 4-5. 시즌
- `season.year`: 4자리 연도. 0이면 Carry Over (시즌 무관)
- `season.season`: `S/S` / `F/W` / `Special` / `Carry Over`

### 4-6. 카테고리 분류
**대분류 (`category`)** — 기본 9개:
`Outerwear`, `Tops`, `Pants`, `Bottoms`, `Skirts`, `Dresses`, `Shoes`, `Bags`, `Accessories`

**중분류 (`sub_category`)** — 자유 형식 (예: `Windbreaker`, `T-Shirt`, `Knit`)

### 4-7. 컬러 / 사이즈 옵션
- `colors_list`: 컬러 옵션 배열. 없으면 `color` 필드의 콤마 분리값 사용
- `sizes`: 사이즈 배열. 단일 사이즈 의류는 `["FREE"]`
- 운영 권장: 컬러별 별도 이미지/SKU 매핑 가능하도록 확장 (현재는 단순 배열)

### 4-8. 이미지
- 시안에서는 base64 또는 외부 URL 혼재
- 운영 권장: S3/Cloudinary 등 통한 URL 통일
- 파트너 포털 카탈로그: 첫 번째 이미지를 썸네일로 사용
- 상품 상세: 모든 이미지 슬라이드/리스트 표시

## 5. 인증 / 권한

### 5-1. 어드민
- 단일/다중 계정 가능 (시안은 단일)
- 모든 데이터 R/W
- 운영 권장: role 분리 (`super_admin` / `operator` / `viewer`)

### 5-2. 파트너
- 본인 데이터만 접근:
  - 카탈로그: visible 상품 R
  - 본인 정보 R
  - 본인 주문 R/W (생성, 취소)
- 다른 파트너의 정보/주문 접근 금지

### 5-3. 비밀번호
- 시안: 미구현 또는 단순 비교
- 운영: bcrypt/argon2 해시, 강도 검증, 변경 기능

### 5-4. 세션
- JWT 또는 세션 쿠키
- 만료 시간 권장: 8시간 (어드민), 24시간 (파트너) — 정책에 따라 조정

## 6. 전시 설정

### 6-1. 카테고리별 노출 순서
- 어드민이 카테고리 안에서 상품 순서를 직접 정렬 (드래그)
- 정의되지 않은 상품은 기본 순서 (등록일 역순 등)

### 6-2. NEW 상품 / HERO 상품
- 메인 페이지 NEW / HERO 섹션에 노출할 상품 선정 (각 최대 5개 권장)
- 어드민이 명시적으로 선택

### 6-3. 메인 배너 (5종)
- `male`, `female`, `kids`, `accessories`, `special`
- 각각: image / title / subtitle / link
- 클릭 시 link로 이동 (필터 적용된 카탈로그 페이지)

## 7. 엑셀 처리

### 7-1. 상품 일괄 업로드
- 양식: 어드민 페이지에서 양식 다운로드 가능
- 컬럼: 스타일 번호 / 상품명(KR/EN) / 카테고리 / ... / 이미지 URL (쉼표 구분)
- 검증 결과를 row 단위로 반환 (성공 N건, 실패 N건 + 에러 사유)

### 7-2. 상품 정보 다운로드
- 현재 필터링된 상품 목록을 엑셀로 다운로드
- 이미지는 클릭 가능한 하이퍼링크로 삽입

### 7-3. 주문 엑셀 다운로드
- 단일 주문: 주문 정보 + 사이즈 타입 매트릭스 형태
- 일괄: 여러 주문 시트 또는 통합 시트
- 한국소매가, 환율, 홀세일가 모두 포함

### 7-4. 주문 일괄 업로드
- 양식: 파트너 / 상품 / 컬러 / 사이즈 / 수량
- 검증: 파트너 존재 / 상품 존재 / 옵션 매칭 / 수량 양수

---

## 8. 검색 / 필터

### 8-1. 상품 검색
- 검색 대상: `name` (부분 일치), `style_no` (부분 일치)
- 대소문자 구분 없음
- 카테고리, 성별, 시즌, 노출여부, HERO 등 다중 필터

### 8-2. 주문 검색
- 검색 대상: `order_id`, `partner_name`
- 대소문자 구분 없음
- 상태 / 파트너 / 기간 다중 필터

### 8-3. 정렬
- 주문: `created_at desc` 기본
- 상품: 등록일 역순 또는 ID 역순 기본

## 9. UX / 운영 정책

### 9-1. 알림 / 토스트
- 승인/반려 처리 시 어드민 화면에 토스트 알림
- 파트너에게 이메일 발송 (현재는 dispatchOrderEmail 함수 stub)

### 9-2. 이메일 발송 (운영 추가 필요)
- 주문 승인 시: 파트너에게 승인 안내 + 인보이스
- 주문 반려 시: 반려 사유 + 제안안 (있으면)
- 주문 생성 시: 어드민에게 신규 주문 알림 (선택)

### 9-3. 다국어
- 시안: 한/영 혼재 (어드민 한국어, 파트너 영어 위주)
- 운영: i18n 적용 권장 (한/영/중/일 등)

### 9-4. 화폐 표기
- KRW: `₩89,000` (천 단위 콤마)
- USD: `$46.14` (소수점 둘째)

### 9-5. 시간대
- 시안: 브라우저 로컬
- 운영 권장: DB는 UTC 저장, 표시는 사용자 시간대 (어드민은 KST 기본)

## 10. 데이터 무결성

### 10-1. 주문 → 상품
- 주문 안의 productId는 해당 시점의 상품을 참조
- 상품 삭제 시: 기존 주문은 영향 없음 (스냅샷 보존)
- 운영 권장: 주문 item 안에 product 스냅샷 (이름, style_no, 이미지) 보존

### 10-2. 주문 → 파트너
- partner_id, partner_name 모두 저장 (denormalized)
- 파트너 삭제/이름 변경 시 기존 주문 영향 없음

### 10-3. 동시 편집
- 주문 승인 처리 중 다른 어드민이 동일 주문 처리 시도 시 → 낙관적 잠금 또는 status 체크 후 거절
- 권장: `If-Match` 헤더로 ETag 또는 `updated_at` 비교
